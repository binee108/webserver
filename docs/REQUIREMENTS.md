# 프로젝트 요구사항 대비 구현 현황

다음 표는 요청받은 핵심 요구사항을 정리하고, 현재 코드가 충족하는 정도(✅ 구현, ⚠️ 부분 구현, ❌ 미구현)를 간략히 표기한 것입니다. 상태가 ⚠️/❌인 항목은 아래 상세 설명과 개선 권고안을 참고하십시오.

| 구분 | 요구사항 | 현황 |
| --- | --- | --- |
| 거래소 확장성 | 다중 거래소 지원 및 용이한 확장 구조 | ⚠️ (Factory 존재, 실 구현은 Binance 위주) |
| 전략 마켓 분리 | 전략이 현물/선물 마켓을 구분해서 동작 | ✅ (`MarketType` 상수와 전략 속성으로 구분) |
| 웹훅 주문 | 단일/배치 주문, 전체 취소, 단일 심볼 취소 처리 | ⚠️ (단일/배치/전체 취소 지원, 단일 심볼 취소 미구현) |
| Rate Limit | 거래소 API Rate Limit 초과 시 딜레이 후 재시도 | ✅ (요청 슬롯 획득 방식으로 대기 후 실행) |
| 주문 우선순위 | MARKET 주문 최우선, 그 외 후순위 | ❌ (현재 요청 순서대로 실행, 우선순위 로직 없음) |
| 전략-계좌 관계 | 전략 하나에 여러 계좌 순차 처리 | ❌ (현재 첫 번째 계좌만 사용) |
| 계좌-전략 관계 | 하나의 계좌를 여러 전략과 연결 가능 | ⚠️ (모델상 가능하나 자산/순차 처리 미흡) |
| 자산 가중 배분 | 계좌 자산을 가중치 비율로 전략별 할당 | ❌ (고정 잔고 값 사용, 실제 자산 기반 아님) |
| 리밸런싱 | 모든 전략 포지션이 없을 때 자산 재배분 | ❌ (재배분 트리거 없음) |
| 전략 손익 반영 | 전략별 손익이 해당 전략 자산만 증감 | ❌ (손익이 자본 배분에 반영되지 않음) |
| 레버리지 적용 | 요청 수량에 레버리지 적용 후 주문 | ❌ (현재 수량 그대로 사용) |
| 전략 공유 | 전략 공개/비공개 및 공유 리스트/구독 | ✅ (is_public 플래그, 구독/해제 API) |
| 수익률 계산 | 전략 실행 거래만으로 수익률 산출 | ⚠️ (기초 통계만 제공, ROI 계산 미흡) |

## 1. 거래소 확장성
- `app/services/exchange.py`의 factory/adapter 구조를 통해 확장 포인트는 존재하나, 실제 구현체는 Binance 위주입니다.
- **개선 제안**: CCXT 래퍼 또는 공통 인터페이스를 보강하고, Bybit/OKX용 어댑터를 추가해야 합니다.

## 2. 전략/마켓 구조
- `Strategy.market_type` 및 `MarketType` 상수를 통해 현물/선물 전략이 구분되며, 자본 배분 루틴에서 마켓 타입별 리스트를 나누고 있습니다.

## 3. 웹훅 주문 처리
- 단일 주문/배치 주문/전체 취소는 지원합니다.
- **부족한 부분**: “단일 심볼 주문 취소” 기능이 구현되어 있지 않습니다. `webhook_service.process_cancel_all_orders`에 심볼별 분기 또는 별도 엔드포인트 추가가 필요합니다.

## 4. Rate Limit 관리
- `RateLimiter.acquire_slot`으로 요청을 대기시킨 뒤 실행하도록 변경하여 요구사항을 충족합니다.
- 일반/주문 엔드포인트를 분리해 초당/분당 제한을 모두 고려합니다.

## 5. 주문 우선순위
- 현재 주문은 입력 순서대로 처리하며, MARKET 주문을 별도로 우선시키지 않습니다.
- **개선 제안**: 배치 주문 등에서 MARKET 주문을 선별해 먼저 실행하는 로직 추가가 필요합니다.

## 6. 전략-계좌 관계 및 순차 처리
- 모델은 한 전략에 여러 계좌를 연결할 수 있지만, `TradingService.execute_trade`가 첫 번째 연결 계좌만 사용합니다.
- **개선 제안**: 전략-계좌 목록을 연동 순서(예: `StrategyAccount.id`)대로 순회하며 각 계좌 별로 주문 시도 후 결과를 모으는 구조가 필요합니다.

## 7. 계좌-전략 가중치 기반 자산 배분
- 요구사항은 계좌 전체 자산을 전략 가중치 합으로 나눠 전략별 할당금을 계산해야 합니다.
- 현 구현은 개발 편의 위해 고정 잔고(예: 10,000)로 계산하고 있어 실제 자산 기반이 아닙니다.
- **개선 제안**: 최신 잔고 스냅샷(예: `DailyAccountSummary`)을 사용해 총 자산을 계산하고, 전략별 가중치로 나눠 `StrategyCapital`에 저장해야 합니다.

## 8. 리밸런싱 및 손익 반영
- 전략에 포지션이 없을 때 자산을 재배분하거나 전략 손익을 해당 전략 자산에만 반영하는 로직이 없습니다.
- **개선 제안**: 포지션 상태를 감시해 리밸런싱을 트리거하고, 거래 손익을 `StrategyCapital`에 누적하는 기능이 필요합니다.

## 9. 레버리지 적용
- 주문 수량에 계좌/전략의 레버리지를 적용하는 단계가 없습니다.
- **개선 제안**: 웹훅에서 전달된 수량에 `StrategyAccount.leverage`를 곱해 `quantity`를 보정한 뒤 주문하도록 변경해야 합니다.

## 10. 전략 공유 및 구독
- `is_public` 플래그로 공개/비공개 전략을 구분하고, 공개 전략은 구독 리스트에서 노출 및 계좌 연결이 가능합니다.
- 비공개 전략은 다른 사용자가 볼 수 없도록 처리되어 있습니다.

## 11. 전략 수익률 계산
- 대시보드와 통계 API에서 기본적인 PnL 합계를 제공하지만, 전략별 ROI(손익/투입자본) 산출, 거래소 직접 거래 제외 등은 구현되어 있지 않습니다.
- **개선 제안**: `Trade` 기록을 활용해 전략별/계좌별 ROI를 계산하고 노출하는 API/뷰를 추가해야 합니다.
